CLI dados

Para criar tópicos, gerenciar schemas, fazer replay, inspecionar lag de consumidores. Funciona em macOS, Linux e WSL.

Comandos principais

  • dados topic create — registra novo tópico no catálogo
  • dados schema register — versiona contrato
  • dados replay — reprocessa janela histórica
  • dados lag — diagnostica consumidor atrasado
  • dados auth login — SSO via OIDC corporativo

Instalação

  • macOS — brew install itau/tap/dados
  • Linux — apt install dados-cli (repo interno)
  • WSL — mesma via apt do Linux
  • Windows nativo — não suportado oficialmente

Governança

Política, SLOs
e trilha de auditoria.

A camada de garantias da plataforma. Schemas versionados em CI, SLOs publicados por domínio, trilha de auditoria para quem leu, escreveu ou alterou cada tópico.

Quatro princípios que não voltam atrás

Princípio 1

Schema-first

Todo dado entra e sai por um schema versionado. Quebras são bloqueadas em CI, não em produção. Compatibilidade backward é o default.

Princípio 2

Event-driven por padrão

Não existe "batch noturno" como caminho feliz. Mudanças propagam em segundos via streams. Batch vira caso especial.

Princípio 3

Multi-região nativo

Latência regional é resolvida no design, não no SLA. Dados se replicam entre SP, RJ e DF conforme política do domínio.

Princípio 4

Self-service com guard-rails

Times criam tópicos sem ticket. Limites, RBAC e validação de schema garantem que self-service não vira caos.

Evolução da cobertura de schemas

SLOs por categoria de domínio

Cada domínio se classifica em uma das três categorias abaixo, definindo SLOs mínimos e cadência de revisão. Esses números são revisados trimestralmente em conjunto com o time de Risco Operacional.

Categoria Domínios SLO disponibilidade SLO latência p99 RPO Auditoria
Tempo real crítico Fraude, Pagamentos 99,99% 100 ms 0 s Mandatória
Transacional Risco, Cadastro, Crédito 99,95% 500 ms 5 s Recomendada
Analítico Investimentos, Atendimento, Compliance, Operacional 99,9% 2000 ms 5 min Opt-in

RPO (Recovery Point Objective) = quanto de dado podemos perder em caso de falha catastrófica. Auditoria mandatória = trilha de leitura também é registrada, não só escrita.

Como mudanças de política são decididas

  1. Etapa 1

    RFC aberta no repositório público interno

    Qualquer engenheiro do banco pode abrir RFC em plat-dados/rfcs. Template pronto, sem aprovação prévia.

  2. Etapa 2

    Comentários e revisão (≥ 7 dias)

    Período mínimo aberto para qualquer time comentar. Itens críticos para Risco/Compliance têm revisão obrigatória dessas áreas.

  3. Etapa 3

    Decisão do comitê quinzenal

    Comitê de 5 pessoas: 2 da Plataforma, 1 de Risco Op, 1 de Arquitetura, 1 rotativo (time consumidor diferente a cada ciclo).

  4. Etapa 4

    Publicação + janela de implementação

    RFC aprovada vira política. Janela mínima de 60 dias antes de virar obrigatória — tempo para times adaptarem.

Quatro canais oficiais. Use o certo para o tipo de demanda — isso ajuda a manter o on-call dormindo à noite quando não há incidente.

Canais e SLA

Em ordem do mais informal ao mais formal. Todos têm dono e tempo de resposta declarado.

slack

#plat-dados — conversa aberta

Para perguntas rápidas, dúvidas de uso, ideias. Sem SLA — respondemos quando podemos, em geral no mesmo dia útil.

→ slack

#plat-dados-suporte — suporte estruturado

Para dúvidas de uso, ajuda em integração, code review. SLA: primeira resposta em 4 horas úteis.

→ slack

#plat-dados-oncall — incidentes em produção

Apenas para incidentes ativos afetando produção. SLA: primeiro contato em 15 minutos, 24/7.

→ repo

plat-dados/issues — bugs e features

Para registrar bug com reprodução, propor feature, abrir RFC. Triagem semanal. Priorização discutida no comitê.

→

Office hours

Toda quinta-feira, 16h–17h, dois engenheiros da Plataforma ficam disponíveis em videochamada aberta para qualquer squad. Sem pauta — chega quem quer, traz o problema, sai com encaminhamento.

Link no canal #plat-dados à 15h55 toda quinta. Gravações ficam em plat-dados/office-hours com índice por tema (não por data).

Enviar uma mensagem direta

Para casos que não se encaixam nos canais acima — parcerias institucionais, demandas regulatórias, casos sensíveis. Esse formulário envia para plat-dados@itau-unibanco.com.br e é lido pela liderança da Plataforma. Não use para incidentes (não tem SLA).

Use seu e-mail @itau ou @itau-unibanco.

Mensagem registrada. A liderança da Plataforma responde em até 3 dias úteis.
