Plataforma de Dados

Handbook do time. Aqui você encontra o que a plataforma é, quando ela é a escolha certa, e como começar a usar — em ordem de profundidade crescente. Para dúvidas pontuais, consulte o FAQ.

O que é

A Plataforma de Dados é a infraestrutura compartilhada que substitui o pipeline batch legado por um modelo event-driven multi-região. Times consomem e produzem dados via contratos versionados, sem precisar operar a infraestrutura.

Pense nela como o "Kubernetes dos dados": ela abstrai a camada de infraestrutura e expõe primitivas estáveis (tópicos, schemas, jobs, materializações) que times de produto combinam para resolver problemas de negócio.

i

Quem mantém

O time de Plataforma de Dados (8 engenheiros) é o owner. Squads de produto são consumidores e produtores — não operam infraestrutura.

Princípios

Quatro decisões arquiteturais que orientam toda evolução da plataforma:

1. Schema-first

Todo dado que entra ou sai da plataforma passa por um schema versionado em Avro. Quebras de compatibilidade são bloqueadas em CI — não em produção. Consumidores antigos continuam funcionando enquanto novas versões coexistem.

2. Event-driven por padrão

Não existe "batch noturno" na plataforma. Mudanças propagam em segundos via streams. Batch é caso especial — uma materialização agendada — não o caso geral.

3. Multi-região nativo

Latência regional é resolvida no design, não no SLA. Dados se replicam automaticamente entre SP, RJ e DF conforme a política de cada tópico.

4. Self-service com guard-rails

Times criam tópicos, schemas e jobs sem pedir ticket. As guard-rails (limites de throughput, políticas de retenção, RBAC) garantem que self-service não vira self-destruction.

Quando usar (e quando não)

A plataforma resolve um conjunto específico de problemas. Para tudo fora desse conjunto, ela é a ferramenta errada.

✓ Use quando

Dado precisa fluir entre 3+ sistemas, ou ser consumido em tempo real, ou compor um produto analítico, ou exigir auditoria de schema.

✗ Não use quando

Dado fica num único sistema, é puramente operacional (config, sessão de usuário), ou tem cardinalidade muito alta (event sourcing por dispositivo).

!

Antes de migrar legado

Migrações de pipelines existentes precisam de aprovação técnica do time de Plataforma. Não é processo burocrático — é alinhamento de capacity e contrato. Abra um RFC antes de começar.

Começando

O caminho recomendado para qualquer time novo é:

  1. Instale o SDK no seu projeto.
  2. Autentique via SSO corporativo.
  3. Liste os tópicos disponíveis na sua área.
  4. Consuma um deles em modo read-only para validar o setup.
  5. Crie seu primeiro tópico (com schema) seguindo o guia abaixo.

Contratos de dados

Schemas são contratos. Eles definem o que produtores garantem e o que consumidores podem assumir.

Compatibilidade

Três níveis aceitos. Toda mudança de schema é classificada e bloqueada se não for compatível na direção esperada:

TipoO que mudaPermitido por padrão
BackwardAdiciona campo opcional, remove campo opcionalSim
ForwardAdiciona enum, renomeia tipoSim, via RFC
BreakingRemove campo obrigatório, muda tipoNão — exige novo schema (v2)

Observabilidade

Todo tópico expõe métricas padronizadas em Prometheus. Painéis prontos no Grafana cobrem throughput, lag de consumo, taxa de erro de schema e latência p50/p95/p99.

Para investigar um problema, comece pelo painel dados-overview filtrado pela sua área.

Incidentes

Em caso de incidente que afete produção e tenha origem na plataforma, acione o time on-call via #plat-dados-oncall. Para suporte não-urgente, use #plat-dados-suporte.

!

Acionamento on-call

O on-call existe para incidentes em produção. Para dúvidas, troubleshoot de dev/staging ou suporte em horário comercial, use o canal de suporte.

FAQ

Posso usar a plataforma para dados pessoais (LGPD)?

Sim, com tópicos marcados como pii: true no schema. Isso ativa criptografia em repouso, mascaramento automático em logs e auditoria de acesso. Toda área que produz/consome PII precisa de aprovação prévia do DPO.

Qual o limite de throughput por tópico?

Limite padrão: 10k eventos/s. Aumentos sob demanda via RFC — sem custo para o time, mas com aprovação técnica para garantir capacity multi-tenant.

Como faço replay de eventos antigos?

O comando dados replay --topic=X --from=2025-09-01 reprocessa um intervalo. Replay é caro — só funciona dentro da janela de retenção do tópico (default 90 dias).

Posso ter um tópico privado, visível só para minha squad?

Sim. No registro do schema, defina visibility: private e liste as squads autorizadas em readers/writers. Visibilidade pública (default) facilita reuso entre times.

A plataforma substitui o data warehouse?

Não. Ela alimenta o warehouse via conector oficial, mas não substitui consultas analíticas em SQL. Pense: plataforma para fluxo entre sistemas, warehouse para análise.

Glossário

Tópico
Stream nomeado de eventos com um schema associado. Unidade básica da plataforma.
Schema
Definição estrutural de um evento, versionada. Garante contrato entre produtores e consumidores.
Materialização
Visão derivada (tabela, índice, agregação) calculada a partir de um ou mais tópicos.
Replay
Reprocessamento de eventos passados dentro da janela de retenção do tópico.
