Onboarding

Seu primeiro tópico em 15 minutos

O caminho recomendado para qualquer time novo. Esses 5 passos colocam você operando em produção em uma sessão.

  1. Instalar o SDK no projeto: npm install @itau/dados-sdk
  2. Autenticar via SSO corporativo: dados auth login
  3. Listar tópicos disponíveis na sua área para confirmar o setup
  4. Definir um schema YAML para o seu primeiro tópico
  5. Registrar e publicar o primeiro evento

Onboarding

Conceitos fundamentais

Vocabulário compartilhado por todos os times que usam a plataforma. Vale a leitura antes de qualquer integração.

  • Tópico: stream nomeado de eventos com schema associado.
  • Schema: contrato versionado em Avro que define a estrutura de eventos.
  • Materialização: visão derivada (tabela, índice, agregação) calculada a partir de tópicos.
  • Replay: reprocessamento de eventos passados dentro da janela de retenção.
  • Conector: ponte entre a plataforma e sistemas externos (warehouse, S3, DBs).

Práticas

Design de schemas

Boas práticas extraídas de 18 meses de uso real. Aplicar evita 80% das dores de evolução.

  • Nomes em snake_case consistentes entre tópicos (cliente_id, não idCliente).
  • Campos novos sempre opcionais — facilita evolução backward-compatible.
  • Valores monetários em long de centavos (valor_cents), nunca float.
  • Timestamps em UTC, em timestamp ISO 8601.
  • Enums fechados quando o domínio é estável, strings livres quando ele evolui.

Práticas

Versionamento e compatibilidade

Schema é contrato. Versionar bem evita comunicação dolorosa com consumidores quando o domínio muda.

  • Minor: adicionar campo opcional, adicionar enum value, frouxar restrição.
  • Major (v2): remover campo obrigatório, mudar tipo de um campo, renomear campo.
  • Versões coexistem por padrão. Consumidores migram no próprio ritmo.
  • Schemas com 0 consumidores podem ser depreciados após aviso.

Práticas

Testes de contrato

Compatibilidade não pode ser garantida no review humano. Use o CI.

  • Cada PR roda dados schemas validate contra o registry atual.
  • Mudanças breaking são bloqueadas no PR, com explicação do que mudou.
  • Consumidores podem registrar "expectativas" — campos que usam ativamente.
  • Mudanças que afetam expectativas geram alerta nos owners dos consumidores.
